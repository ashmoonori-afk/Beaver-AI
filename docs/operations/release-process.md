## TL;DR

```bash
# Bump versions in package.json, packages/desktop/package.json,
# and packages/desktop/src-tauri/{Cargo.toml,tauri.conf.json}.
git tag v0.1.0
git push origin v0.1.0
```

The `release` workflow runs on tag push, builds the desktop app, and uploads artifacts to a draft GitHub Release at https://github.com/ashmoonori-afk/Beaver-AI-Dev/releases.

## Per-OS bundles

Phase 4D.3 ships Windows only. Phase 4D.4 adds the matrix:

| OS host (CI runner) | Targets emitted |
| ------------------- | --------------- |
| `windows-latest`    | NSIS `*-setup.exe`, MSI `*_x64_en-US.msi` |
| `ubuntu-22.04`      | `*.deb`, `*.AppImage` (4D.4) |
| `macos-13`          | `*.dmg` (4D.4; notarization opt-in) |

## Code signing (opt-in)

Set these in repo Settings → Secrets:

| Secret name | Purpose |
| --- | --- |
| `TAURI_SIGNING_PRIVATE_KEY` | Self-signed updater key (per 4D.0). Passed as env var to `tauri build`. |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | Optional passphrase for the key. |
| `APPLE_CERTIFICATE` | macOS notarization (4D.4 only). |
| `APPLE_CERTIFICATE_PASSWORD` | Same. |
| `APPLE_ID` / `APPLE_PASSWORD` / `APPLE_TEAM_ID` | Notarization auth. |

When secrets are absent, the build still succeeds — it produces unsigned artifacts. For our self-signed channel that's acceptable; users get an OS warning on first launch.

## Generating a self-signed updater key

Local one-off (do not check the resulting key into git):

```bash
pnpm --filter @beaver-ai/desktop tauri signer generate -w ~/.tauri/beaver.key
# Public key prints to stdout — paste into tauri.conf.json updater.pubkey.
# Private key contents go into the TAURI_SIGNING_PRIVATE_KEY secret.
```

## Release notes

Auto-generated by the workflow from the git log between the previous tag and the current tag. Edit the draft release on GitHub before publishing.

## Smoke test before publish

1. Download the draft release artifacts.
2. Install the MSI on a clean VM.
3. Launch the .exe — Bento UI must render with the demo state machine.
4. (Once 4D.2.x lands) submit a real goal — verify a CLI sidecar spawns and bento updates from real events.
5. Edit the draft release → Publish.
