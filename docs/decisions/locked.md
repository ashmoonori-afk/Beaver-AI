# Locked Decisions (D1–D16)

> Foundational and second-tier decisions that the rest of the architecture is built on. Each entry locks a single choice and records the reason.

**Doc type:** decisions
**Status:** Locked
**Last updated:** 2026-04-26 (D16 added)
**See also:** [decisions/open-questions.md](open-questions.md), [overview.md](../overview.md)

---

| #  | Decision | Choice | Rationale |
|----|----------|--------|-----------|
| D1 | Language / runtime | **TypeScript on Node.js (≥ 20 LTS)** | First-class SDKs for all major LLM providers; target CLIs (Claude Code, Codex) are themselves Node-based, making `child_process` integration native; strong typing for tool schemas; single npm package can ship both the library and the CLI. |
| D2 | Distribution shape | **Library + CLI (dual entry, single package)** | One npm package (`beaver-ai`) exposes both `import { Beaver }` and a `beaver` binary. State per project lives on the filesystem under `.beaver/`; no daemon required. See [architecture/entry-layer.md](../architecture/entry-layer.md). |
| D3 | External LLM integration | **Spawn external CLIs as child processes** | Reuse the strengths of existing tools (Claude Code's file edit + sandbox, Codex's strengths) rather than re-implement. Adapter layer normalizes the interface. Direct API calls remain available for lightweight in-process tasks. See [architecture/provider-adapters.md](../architecture/provider-adapters.md). |
| D4 | Workspace & state | **Git worktrees + SQLite** | Each agent runs in an isolated worktree on its own branch — physical isolation gives natural conflict handling and rollback. SQLite holds the run/task/event ledger for queryability and resumability. See [architecture/workspace-state.md](../architecture/workspace-state.md). |
| D5 | Feedback / interrupt UX | **Terminal interactive checkpoints + Web dashboard (dual)** | Both backed by the same checkpoint primitive. Synchronous CLI usage gets blocking prompts; long autonomous runs get a dashboard with approve / reject / edit affordances. See [architecture/feedback-channel.md](../architecture/feedback-channel.md). |
| D6 | Orchestrator decision model | **Fixed top-level FSM + LLM-driven sub-decisions** | Top-level transitions are deterministic code (observable, resumable, debuggable). Within each state, judgment calls are LLM calls against structured prompts. No external graph framework — Beaver is itself the orchestrator. See [architecture/orchestrator.md](../architecture/orchestrator.md). |
| D7 | Cost / budget primitive | **USD ground truth · 3-layer caps (agent / task / run) · hard cap → checkpoint** | Users care about USD, not tokens; rate tables convert internally. Three nested caps prevent both runaway agents and runaway runs. Hard cap triggers a `budget-exceeded` checkpoint with stop / increase / continue-once. See [models/cost-budget.md](../models/cost-budget.md). |
| D8 | Plan format | **JSON DAG · zod schema · monotonically versioned** | Machine-readable for runtime, diff-friendly for reviewers, validatable against a single source-of-truth schema. Plans are immutable per version; mutations produce a new version with a `parent_version` link. Markdown rendering is derived. See [models/plan-format.md](../models/plan-format.md). |
| D9 | Sandbox & shell policy | **Worktree write boundary · pre-tool-use policy hook · `risky-change-confirmation` with indefinite pause · network allow + audit** | Autonomy is bounded by an explicit policy: writes outside the worktree, broad-blast-radius patterns, and supply-chain risks pause for user approval; a small hard-deny set (sudo, system destruction, credential paths) auto-aborts; network is allowed but every destination is logged. OS-level sandboxing is deferred to v0.2 hardening. See [models/sandbox-policy.md](../models/sandbox-policy.md). |
| D10 | Agent operations policy | **Bounded parallel (default 5) · max 2 retries per task · CLI-only providers (Claude Code default; Codex for `coder`; `integrator` deferred past v0.1) · fresh context with structured dependency summaries · per-role wall-clock + 120 s output-stall watchdog · OS-default resource limits (rlimits deferred to v0.2)** | Bounded concurrency keeps cost variance and transcripts manageable. Failure-typed retry policy avoids burning budget on non-recoverable errors. CLI-only prototype gives a single integration story to debug end-to-end; orchestrator sub-decisions also use Claude Code CLI for consistency. Codex is reserved for code-heavy roles to validate its sandbox PATH-shim path early. Stall watchdog catches hangs early without OS-level instrumentation; explicit resource limits wait for the v0.2 OS sandbox so legitimate builds are not artificially constrained. See [models/agent-operations.md](../models/agent-operations.md). |
| D11 | UX flow | **Free-text goal + one planner clarification pass · approve/comment/reject on plan · Ctrl-C single = graceful pause, double = hard kill · final-review leaves branches in place (no auto-merge) · one active run per project** | Beaver behaves as an autonomous colleague: it asks for what it needs, accepts amendments by sentence, never silently puts code into the user's main branch, and never multiplexes confusing parallel runs. Auto-merge / PR / multi-run / templates are deferred to v0.2 where a dashboard can support them safely. See [models/ux-flow.md](../models/ux-flow.md). |
| D12 | UI policy (CLI surface) | **Normal verbosity default · bottom-fixed live status line · compact-list plan rendering · semantic colors paired with text/symbol · terse no-emoji tone · pretty in TTY, plain when piped** | Governs the **secondary CLI surface**; the **primary web UI** is governed by [models/app-ui.md](../models/app-ui.md) (D13). For the CLI: autonomous tool's bigger sin is over-output, so default suppresses agent stdout streaming. A single status line gives glanceable progress. Plan compact-list is densest for 5–10-task plans. Color decorative-only; every status also conveyed by symbol/text for color-blind safety and pipe-friendly output. See [models/ui-policy.md](../models/ui-policy.md). |
| D13 | App UI strategy | **Hybrid v0.1: localhost web app is the primary user-facing surface; CLI is preserved as scripting/headless secondary. `beaver run` auto-launches a browser at `http://127.0.0.1:<port>/runs/<id>`. Server lives while run is RUNNING/PAUSED, lingers 60 s after terminal state. Localhost-only, no auth, bound to `127.0.0.1`. Goal entry stays on CLI; everything the web UI shows is reachable from CLI subcommands.** | Beaver's product pitch is "leave it running, come back to check" — that fits a GUI better than a terminal. Hybrid preserves the CLI invariants (D2) and the headless workflow, while adding a thin renderer layer over the SQLite ledger. Localhost-only bind keeps the trust model identical to common dev-tool servers (Vite, Storybook). Q4 (dashboard auth) resolves here. Tauri/desktop packaging is deferred to v0.2 — the same web bundle is wrappable later. See [models/app-ui.md](../models/app-ui.md). |
| D14 | Workflow personalization & Wiki system | **4-tier last-wins config (defaults → user OS-conventional → project → flags) · coding style implicit via worktree access · learning delivered by an LLM-maintained Wiki system at `<config>/wiki/` (suggest-only — never auto-applied) · no goal templates** | Personalization is layered transparently so per-machine prefs (display, default budgets, concurrency) live across projects without re-typing. Coding style stays in the repo's own config files — Beaver does not fork conventions. Learning is implemented as a *living wiki* (not a flat memory log): Beaver itself ingests after every run, queries before every plan and risky-change checkpoint, and the wiki compounds across runs as Memex-style structured markdown. Goal templates and `beaver wiki lint` are deferred to v0.2. See [models/personalization.md](../models/personalization.md) and [models/wiki-system.md](../models/wiki-system.md). |
| D15 | Agent behavioral baseline | **Bundled `AGENT_BASELINE.md` injected as the first layer of every agent's system prompt: 4 principles — Think Before Coding · Simplicity First · Surgical Changes · Goal-Driven Execution. Provider-agnostic source, dual-named on disk: `CLAUDE.md` for Claude Code agents, `AGENTS.md` for Codex agents. Additively merged with the repo's `CLAUDE.md` and/or `AGENTS.md` (if present) plus a short per-role addendum, then the task prompt.** | Without a stable posture, autonomous agents drift toward over-engineering, speculative refactors, and silent assumptions — the failure modes that erode trust. The baseline is short, opinionated, ships with Beaver, and applies regardless of provider. The single source is rendered under whichever filename the spawning CLI auto-discovers, so the same canonical content reaches Claude Code and Codex agents alike. Merging is additive so user repos with their own conventions add to rather than override the baseline. See [models/agent-baseline.md](../models/agent-baseline.md). |
| D16 | App UI tech stack | **Server: Fastify (TS, Node 20) · Frontend: React + Vite (TS) · Styling: Tailwind + shadcn/ui (dark default) · Live updates: Server-Sent Events with 3-second polling fallback on disconnect · Routing: single-page with hash routes (`#status`, `#checkpoints`, `#plan`, `#logs`, `#review`) · Browser fallback: auto-detect graphical environment, otherwise print URL + `--no-server` guidance** | Choices target the v0.1 hybrid prototype: Fastify is fast cold-start and idiomatic Node; React + Vite is industry standard and trivially wrappable into Tauri later (D13's deferred desktop story). Tailwind + shadcn match D12's "tools-feel" tone — dense, low-chrome, dark default. SSE matches Beaver's near-monodirectional update flow without WebSocket complexity. Hash routes avoid server-side wildcards and make the UI feel like one app rather than navigations. Browser fallback respects SSH / container environments without surprising the user. See [models/app-ui.md](../models/app-ui.md). |
